# ReadMeCraft 产品审计报告 v1

日期: 2026-05-12

---

## 一、核心价值 —— 不够锋利

**问题：** 产品卖点散落在 Hero 页，用户 3 秒内抓不住"为什么要用你"。竞品 readme.so 强在可视化编辑，GitHub 官方强在集成——而 ReadMeCraft 的杀手锏应该是 AI + 中文语境，但这个信息不突出。

**建议：**
- Hero 标题改为 *AI 生成中文 README · 输入仓库地址，一分钟出稿*（把"AI"和"中文"前置）
- 去掉"告别枯燥编写，拥抱高效创作"这类行业套话，换成可感知的承诺：*5 个模板可选 · 自动分析源码 · 所见即所得的编辑*

---

## 二、用户漏斗 —— 从输入到下载，每一步都在流失

**问题：**
1. 生成等待不透明 — 点击生成后 spinner 在转，但用户不知道进度（"正在分析源码… 正在调用 AI…"），5 秒后可能直接放弃
2. 生成后没有"下一步"引导 — 跳转到编辑区后，用户可能茫然：我该编辑什么？怎么导出？虽然有 onboard 提示条但还是不够直观

**建议：**
- 生成过程拆 3 步进度：🔍 分析源码 → 🤖 AI 生成 → ✅ 完成（后端 SSE 推送进度，或前端按阶段伪进度）
- 编辑区首次进入时，高亮下载按钮脉冲动画 + tooltip "点击下载 README"

**备注：** 首页已有 5 个模板预览和 Demo 生成入口，无需额外示例按钮。之前建议的一键填充（一键填充仓库 URL）与用户需要理解输入格式的需求矛盾，已撤回。

---

## 三、AI 输出质量 —— 信任度的瓶颈

**问题：** 用户已经反馈过模板不一致的问题。目前的 validate-retry 机制只修复了结构性缺失（表格行数、mermaid 类型），但语义质量没有保证——AI 可能输出空洞内容但没有触发修正。另外用户无法干预：如果 AI 写偏了，反馈重生成是唯一的交互，但需要用户手动写 prompt。

**建议：**
- 生成后做 README 质量评分（章节完整性 + 内容充实度 + 技术细节数），在编辑区顶部以仪表盘显示（绿色/黄色/红色），让用户知道"这个 README 还不错"还是"建议优化"
- 反馈重生成按钮下方预置常用修正：*内容太简略 / 技术栈描述不准确 / 安装步骤有误 / 需要更多使用示例*，降低用户输入门槛
- 考虑增加"严格模式"开关——打开时骨架结构和内容长度强制约束，适合企业级项目；关闭时让 AI 自由发挥，适合创意型项目

---

## 四、模板系统 —— 概念好但落地浅

**问题：**
- 5 个模板是风格维度，不是场景维度。一个 Web 框架项目和一个 CLI 工具用同一个模板，输出结构一样，差别只有样式
- 没有"智能推荐"——用户选了仓库后，系统应该根据语言/框架/项目类型自动推荐最合适的模板
- minimal 模板对 AI 限制太多（禁止 HTML/表格/emoji/引用块），生成的内容平淡如水，和"极简清风"的设计初衷不是一回事

**建议：**
- 增加场景维度：API 类库模板、CLI 工具模板、前端项目模板、后端服务模板——和现有风格模板交汇
- 获取仓库信息后，自动推荐模板并短期高亮推荐标签（"推荐 ✓"）
- minimal 模板重新定义——"极简"不等于"禁止一切"，而是"排版优雅、信息层次清晰"。允许表格和代码块，但去掉装饰性元素

---

## 五、编辑体验 —— 差最后一公里

**问题：**
- 没有 Markdown 工具栏 — 用户要手动输入 `##` `**粗体**` `- 列表`，门槛高。对标 Notion / Typora 的 toolbar 缺失
- 手机端几乎不可用 — 输入仓库 URL 后在手机上编辑/预览分屏不可用，只能 tab 切换
- 没有"插入图片" — README 经常需要截图/logo/架构图，但编辑器只支持纯文本

**建议：**
- 编辑器上方加浮动 toolbar：加粗、标题 H2/H3、列表、代码块、分割线、引用块（6 个最常用操作）
- 移动端检测到编辑器输入时默认切换到编辑 tab
- 增加图片插入按钮 → 打开 URL 输入框（支持 GitHub 仓库内图片路径识别）

---

## 六、可发现性 —— 功能藏在角落

**问题：**
- 反馈重生成按钮在预览面板底部（用户可能根本滑不到那里）
- 历史记录在 Header 的时钟图标（小且无文字标签）
- 快捷键面板藏在编辑区右下角的"?"（极少用户会注意到）
- 清空数据在 Header 的垃圾桶——入口不错但是无文字说明

**建议：**
- FeedbackCard 移到编辑器底部或作为浮动按钮常驻
- 历史记录在 Header 加文字"历史记录"（当前已有文字，但可以更显眼一些，比如在第一次完成生成后 badge 闪烁一下）
- 用户首次进入编辑区时，右下角 toast 提示"按 ⌘? 查看快捷键"

---

## 七、增长闭环 —— 没有病毒循环

**问题：**
- 生成的 README 底部没有"由 ReadMeCraft 生成"的标识（用户无法给产品带来曝光）
- 没有"分享"功能（一键分享到 Twitter/X、复制分享链接）
- 没有社区画廊——人家用这个工具生成了多好的 README，潜在用户看不到

**建议：**
- 在生成的 README 底部以注释方式加 `<!-- Generated by ReadMeCraft -->`（非侵入，不影响渲染）
- 增加"分享"按钮 → 复制带有项目名和模板名的分享文本
- 首页底部增加「精选 README」展示 3 个示例，点击可编辑/复制

---

## 八、技术产品化

**问题：**
- Toast 组件内联在 App.tsx 中（line 19-66），应该抽成独立组件
- useReducer 的 action 类型已膨胀到 20+，接近需要拆分 context 的临界点
- 前端没有 error boundary（AsyncBoundary.tsx 已存在，但没有正确应用 fallback）
- 后端 analytics.ts 用 NDJSON 纯文本存储，部署后不可靠（容器重启、并发写入都有问题）

**建议：**
- Toast 抽离为独立组件，顺便增加 Toast 队列上限（当前无限追加）
- 考虑将 state 分层：repoContext / editorContext / uiContext，减少无关状态触发的重渲染
- AsyncBoundary 接入每个路由包裹，设置友好 fallback
- 后端分析存储换 SQLite（写时复制、无锁读）或直接扔给下游 log 服务

---

## 优先级建议

**三 > 五 > 二（仅进度透明 + 首次引导）> 七 > 一 > 六 > 八**

说明：先解决 AI 输出质量和模板落地（三是口碑基础），再优化漏斗和编辑体验（二和五决定留存），最后做增长和技术基建。
